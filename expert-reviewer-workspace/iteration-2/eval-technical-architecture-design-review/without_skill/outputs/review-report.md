# 架构评审结论

先给结论：这套方案能跑通一个演示版，但离“双 11 大促可扛压、可回退、可定位、可扩展”的生产级订单系统差得很远。最大的问题不是技术栈选型本身，而是你把分布式系统里最难的几件事全都写成了“调用一下、失败就回滚”这种单机事务思路。这个设计在低并发、低失败率、低峰值时看起来很顺，一旦进入大促、链路抖动、局部故障、消息堆积、库存和余额争抢，就会出现大量不一致、重试风暴、超时雪崩和人工对账地狱。

如果你要我严一点说，这份设计目前更像“把几个常见组件拼起来”的清单，不像一套能承受真实交易压力的系统方案。下面我直接挑刺。

## 一、最大的问题：你把跨服务事务当成了本地回滚

你写的订单流程是：用户下单后，订单服务创建订单，状态先是 `PENDING`，然后调用用户服务扣余额，再调用商品服务扣库存，最后更新订单为 `SUCCESS`；中间失败就回滚。

这句话本身就是整个方案最危险的地方。

原因很直接：订单服务、用户服务、商品服务是三个独立数据库，三个独立服务，网络调用也不是原子操作。你无法用一个“回滚”把已经发出去的远程调用撤掉。尤其是余额扣减和库存扣减，一旦某个步骤成功、后续步骤失败，你就必须处理补偿，而不是幻想事务回滚会替你抹平分布式边界。

更糟糕的是，你没有说明“失败”具体指什么。是接口返回错误？超时？网络抖动？半成功？服务实例挂了但数据库已经提交了？这些场景在大促里会频繁发生。你如果把所有异常都简单归为“失败”，然后尝试回滚，很快就会出现：

- 订单已经扣了库存，但状态还在 `PENDING`
- 用户余额扣了，订单回滚失败，钱出去了订单却没成功
- 第三方超时重试后重复扣减
- 重试和补偿互相打架，最后变成双扣或漏扣

建议很明确：不要把跨服务流程设计成“先做 A，再做 B，失败就回滚 A”。你至少要明确补偿机制、幂等策略、状态机流转和最终一致性路径。更现实的做法是把“下单成功”拆成可恢复的阶段，不要在一个同步链路里要求所有动作一次性完成。

## 二、同步串行调用太重，核心链路会被你自己拖死

当前流程是前端请求进入订单服务后，订单服务同步调用用户服务，再同步调用商品服务，最后同步改订单状态。

这条链路的问题不是“能不能通”，而是“扛不扛得住”。双 11 场景下，最怕的就是长链路同步阻塞。只要用户服务稍慢一点、商品服务稍慢一点、网络抖一下，订单接口 RT 就被拉长，连接池、线程池、网关并发、Tomcat 工作线程会一起吃紧。

你还用了 OpenFeign，这意味着每个远程调用都默认增加一次网络往返和一次序列化开销。如果没有特别严格的超时、重试、隔离、降级设置，系统在高峰期会出现非常典型的级联故障：

- 某个下游慢了，调用方线程堆积
- 线程堆积后，请求排队时间变长
- 请求变长后，客户端超时重试
- 重试进一步放大流量
- 下游更慢，最后整体雪崩

你现在的方案里完全没看到“超时预算”概念，也没看到任何隔离措施。对大促来说，这不是漏项，这是致命漏项。

建议：核心下单链路要尽量短，能异步化的部分尽量异步化。同步链路里只保留必须的校验与关键预占，其他动作通过消息驱动补齐。并且要明确每一跳的超时上限、失败策略、重试上限和熔断条件。

## 三、你现在的库存和余额模型，很容易把“扣减”做成事故源头

余额和库存是最容易出问题的两个资源，因为它们天然高并发、强竞争、强一致诉求高。

如果用户服务负责扣余额，商品服务负责扣库存，你实际上在两个不同的系统里做“资源锁定”。但你没有说：

- 是直接扣减，还是先预占再确认
- 是乐观锁，还是悲观锁
- 是按单冻结，还是按账户实时扣减
- 重复请求如何识别
- 并发下同一用户/同一 SKU 如何避免超卖和超扣

没有这些细节，所谓“扣减”只是一个词，不是方案。

尤其是商品库存，双 11 最常见的问题不是“平均响应慢”，而是热点 SKU 被打爆。你如果每次都去商品库里做强一致更新，数据库行锁会很快成为瓶颈；如果你只靠 Redis 缓存库存，又没有严格的原子扣减和回源校准，最后就会超卖。

余额也一样。只要有超时重试、重复提交、客户端刷新、网关重试、Feign 重试，就可能产生重复扣费。你没有写幂等键、订单号唯一约束、请求去重和补偿核对，这在金融相关扣款里是不可接受的。

建议：余额和库存都不要简单理解成“服务调用成功就算完成”。要引入预扣、冻结、确认、释放的明确状态，配合唯一业务键和数据库约束，做到重复请求不会造成重复变化。

## 四、订单状态设计太粗，无法承载真实流程

你现在只写了 `PENDING` 和 `SUCCESS`，失败就回滚。

这个状态设计过于幼稚。真实订单流程里至少会遇到创建中、待支付、待确认、库存预占成功、余额冻结成功、部分失败、取消中、已取消、超时关闭、补偿中、已完成、退款中、退款成功、退款失败等一堆状态。

你只保留两个状态，会直接导致两个后果：

第一，系统无法表达中间态，排障时只能看日志猜发生了什么。第二，补偿和重试无法挂载到明确状态上，最后每次故障都只能靠人工翻数据库和消息队列。

订单系统最怕的不是复杂，而是复杂被假装成简单。状态不够，最后一定会长出一堆布尔字段、临时标记和“异常订单处理脚本”，这比一开始设计得清楚更难维护。

建议：订单状态要按业务阶段细分，并且每个状态只能单向流转，不能随便回跳。否则你后面无论是补偿、对账还是恢复任务，都会变成代码里的分支地狱。

## 五、你没有说清楚消息队列 Kafka 的角色，等于把它挂成了摆设

方案里写了 Kafka，但订单流程却完全是同步调用。这样 Kafka 到底负责什么？发订单事件？削峰？异步补偿？订单超时关闭？库存回补？支付回调？审计日志？

如果 Kafka 只是“系统里有这个组件”，那它没有架构价值，只是技术标签。

如果 Kafka 真要承担核心异步链路，你就必须回答几个关键问题：

- 订单创建后，哪些事件要发到 Kafka
- 事件发送和本地事务如何保证一致
- 消费端如何幂等
- 失败消息如何重试、如何入死信队列
- 消息积压时如何限流和保护下游
- 消费顺序是否要求按订单维度保持一致

没有这些内容，Kafka 只会在故障时给你增加一层复杂度：数据库已经改了，消息没发出去；消息发出去了，数据库没改；消费者重复消费了；同一订单的消息乱序了；积压后延迟几分钟甚至几十分钟才处理。

建议：把 Kafka 明确成事件驱动的异步骨干，而不是装饰品。至少要说明它承担哪几类事件、在哪些节点生产和消费、如何保证幂等和补偿。

## 六、Redis 也写得太轻，像缓存标题，不像缓存方案

你列了 Redis，但没有说明它缓存什么，怎么失效，怎么防穿透、击穿、雪崩。

双 11 场景下，Redis 不是“有了就快”，而是“没用好就翻车”。

如果缓存商品详情，问题是热 key、失效风暴和缓存一致性。
如果缓存库存，问题是原子性、热点争抢和回源压力。
如果缓存用户会话，问题是登录态膨胀和集群漂移。
如果缓存订单读模型，问题是数据延迟和写后读不一致。

你一个都没说。

特别是库存场景，Redis 如果只是普通 get/set，根本不够。你至少需要说明是否使用原子脚本、是否做库存预热、是否做热点隔离、是否做本地缓存兜底、是否有降级读路径。否则 Redis 只会把数据库压力转移成缓存一致性事故。

建议：不要把 Redis 当万能提速器。要明确它是做热点保护、状态预占、会话承载，还是做消息幂等标记，不同用途的设计完全不同。

## 七、数据库按服务拆库是对的，但你没有处理最关键的读写协同问题

订单一个库、用户一个库、商品一个库，这个方向本身没错。问题在于，你只写了“拆库”，没有写“拆库后怎么协同”。

拆库以后，最麻烦的不是写，而是跨库查询和最终一致性。

例如订单详情页通常要展示订单信息、用户信息、商品信息、支付信息、物流信息。你要么在前端串多个接口，要么在网关聚合，要么做读模型冗余。可你当前方案里没有任何读侧设计。结果就是：写侧拆得很干净，读侧全靠实时调用拼接，慢、脆、难缓存。

再比如报表、对账、运营查询、客服检索，这些需求一定会来。你如果只按业务域拆库，不考虑读模型和数据同步，后面一定会出现一堆“临时 SQL”“人工导出”“离线脚本”来补洞。

建议：明确读写分离后的查询路径。核心页面和高频查询最好有独立的读模型或聚合层，不要临时让前端拼接多个服务。

## 八、Nacos 只写出来不够，你需要考虑配置、注册、发现背后的稳定性

Spring Cloud + Nacos 是常见组合，但问题在于你把它当成“基础设施会自动工作”。不会。

大促期间，注册中心和配置中心本身也会成为脆弱点。服务实例频繁上下线、心跳抖动、配置变更、灰度切换、网络分区，都会影响服务发现的稳定性。只要注册中心出现短暂抖动，下游服务就可能出现实例列表变化、路由异常或短暂不可用。

你没有提：

- 注册中心是否高可用部署
- 配置是否按环境隔离
- 配置变更是否支持回滚
- 服务发现异常时是否有本地缓存兜底
- 实例剔除和重连策略怎么调

这些不写，架构图上有 Nacos，生产里却可能被一个配置误操作搞崩。

建议：把 Nacos 视为生产级关键依赖，明确高可用、变更审计、回滚和默认配置策略，不要只停留在“用了哪个组件”。

## 九、网关只写了 Gateway，没有说保护策略，这会把入口暴露成大口子

Gateway 是入口，但入口不是只做转发。大促期间，网关至少要承担鉴权、限流、黑名单、灰度、路由、降级、请求大小限制、统一超时控制、熔断预警等职责。

你现在没有任何说明。这样一来，前端一旦被脚本刷单、恶意并发、重复提交、接口重放，流量会直接冲击后端订单服务。

尤其是下单接口，如果没有幂等保护和令牌机制，用户重复点击、网络重试、前端刷新都可能生成重复订单。仅靠后端“查一下有没有同样的单”通常不够，因为并发下会发生竞态。

建议：网关层必须有基础防护，订单接口必须有防重复提交机制。这个不是优化项，是底线。

## 十、你完全没提幂等，这是订单系统的大雷

订单、扣款、扣库存、发消息，这几个动作每一个都必须幂等。

但你的方案里一个幂等机制都没出现。

没有幂等，就意味着：

- 客户端重试会造成重复下单
- Feign 超时重试会造成重复扣减
- Kafka 重投会造成重复消费
- 定时补偿任务可能重复执行
- 运维手工重放消息会造成重复变更

订单系统不是一次成功就结束，真正的生产环境里“重复请求”是常态，不是异常。你现在的设计没有这个意识，说明还停留在单次请求的思维。

建议：所有写操作都必须定义业务幂等键、唯一约束或去重表，且要明确幂等窗口和重复返回策略。

## 十一、你没有考虑超时、重试和熔断的组合效应

OpenFeign 默认不是你想象中的“稳定远程调用工具”。真正决定系统稳定性的，是超时、重试、隔离、熔断、降级的组合。

如果超时太长，线程会被拖死；如果超时太短，正常波动也会被判失败；如果重试太激进，瞬时问题会被放大成流量风暴；如果熔断阈值设置不合理，又会让本来还能服务的下游被提前打断。

你现在的设计里一个参数都没有，这意味着评审阶段就已经把稳定性外包给“默认值”。这在生产里通常等于没设计。

建议：为每条调用链定义明确的超时预算和重试边界，并且区分读调用和写调用。写调用通常不适合盲目重试，尤其是涉及库存和余额变更时。

## 十二、没有对大促峰值做容量设计，等于没谈双 11

你说目标是支撑双 11 大促，但整份方案没有任何容量思考。

双 11 不是“用户多一点”，而是流量在很短时间内集中爆发，热点商品高度集中，某些接口的 QPS 会比平时高几个数量级。你如果没有预估：

- 峰值 QPS
- 订单创建吞吐
- 库存扣减峰值
- Kafka 吞吐和积压上限
- Redis 热点 key 压力
- 数据库连接池上限
- 分库分表策略

那就谈不上支撑大促。

目前这个设计是按“正常业务流量”在想问题，不是按“峰值突刺+局部故障”在想问题。大促系统最怕“平均值思维”，因为平均值在真实高峰里毫无意义。

建议：必须先给出峰值指标，再反推架构和组件承载能力。没有容量指标，架构讨论就是空谈。

## 十三、你没有任何失败恢复和对账机制

真实订单系统一定会出现半成功。

例如订单创建成功、扣库存成功、扣余额失败；或者余额成功、库存失败；或者消息发成功但消费者没处理；或者补偿任务执行了一半中断。这些都不是边角料，而是主路径的一部分。

你的方案里没有：

- 失败订单如何自动恢复
- 订单超时如何关闭
- 库存预占如何释放
- 余额冻结如何解冻
- 异常状态如何人工介入
- 账实不一致如何对账修复

没有这些，系统一旦出错，最终只能靠数据库修数据。

建议：把失败恢复和对账当成一等公民，而不是事故后的补丁。订单系统的成熟度，往往不在“正常下单有多顺”，而在“故障后能不能自己回来”。

## 十四、缺少审计、追踪和定位能力，出了问题你很难知道坏在哪

大促期间最怕的是“系统看起来还活着，但就是慢、就是错、就是偶发失败”。这种问题没有可观测性几乎无法排查。

你的设计里没有提到：

- 全链路 traceId
- 订单号贯穿日志
- 关键状态变更审计
- 指标监控
- 消息积压监控
- 下游调用成功率和延迟分布
- 告警阈值和自动止血

如果没有这些，事故发生后你只能靠猜。订单系统不是开发完就结束，真正的成本在故障时的定位和恢复。没有可观测性，系统越大，运维越像盲人摸象。

建议：从第一天就把日志、指标、链路追踪、告警和审计打通，尤其是订单状态变化和资源扣减必须可追溯。

## 十五、系统边界不清，责任划分有问题

从你的描述看，订单服务、用户服务、商品服务都有自己的数据库，但你没有明确它们的职责边界。

例如：

- 余额扣减是用户服务的核心职责，还是订单服务编排时调用的一个能力
- 库存扣减是否允许商品服务独立决策，还是必须依赖订单上下文
- 订单最终状态到底由谁来定，订单服务还是流程编排器

如果边界不清，后面团队协作时就会出现大量扯皮：到底是谁负责补偿，谁负责幂等，谁负责恢复，谁负责对账。

建议：每个服务必须有清晰的领域边界和变更责任。否则拆成微服务之后，只是把单体里的耦合换成网络里的耦合。

## 十六、前端 React 不是问题，但你没有考虑提交体验和防重复控制

React 本身没问题。问题在于订单系统前端不是普通展示页，它会直接影响重复下单和用户感知。

你没提：

- 下单按钮防连点
- 提交中的禁用态
- 重试后的结果展示
- 请求超时后的用户提示
- 重复提交时如何引导用户查单

如果前端体验没设计好，后端幂等就会承受更多压力。前端不是装饰层，它是大促流量控制的一部分。

建议：前端要配合做最基本的去重和状态提示，但不要把安全性完全寄托在前端，因为前端永远可以被绕过。

## 十七、你没有说数据一致性模型，导致所有关键判断都悬空

这套方案里最缺的不是组件，而是明确的一致性模型。

你到底要的是：

- 强一致，还是最终一致
- 余额和库存是否允许短暂不一致
- 订单状态更新延迟是否可接受
- 超时订单和成功订单冲突时谁优先
- 失败补偿是否允许重复执行

这些问题不回答，后面所有实现都没有统一标准。开发会按自己的理解写，测试会按自己的理解测，最后线上故障会按最坏方式出现。

建议：先定义一致性边界，再决定技术方案。不是先堆技术，再期待一致性自然出现。

## 十八、如果按现在这版上线，最可能出现的线上事故

我直接按结果说，最可能出现的是下面几类事故：

第一类，超卖和重复扣费。原因是没有幂等、没有明确资源锁定模型、没有补偿闭环。

第二类，请求堆积和接口雪崩。原因是同步串行调用过长、缺少超时隔离和降级。

第三类，半成功订单大面积堆积。原因是远程调用不可原子，但你把它写成了可回滚。

第四类，消息积压后延迟处理。原因是 Kafka 没有明确职责，消费链路和补偿链路都没定义。

第五类，故障后人工介入频繁。原因是缺少状态机、对账、审计和恢复机制。

这些不是“可能性较低的小问题”，而是这个方案的自然演化结果。

## 最后一句话

如果你想把这个系统真的做成能抗大促的订单平台，最少要把“同步串行扣减 + 失败就回滚”这套思路彻底推翻。要重新定义订单状态、幂等、补偿、消息驱动、库存和余额的冻结确认模型、超时重试边界、可观测性和对账恢复。

现在这份设计，适合做课堂作业，不适合直接做双 11 主链路。
