# 实时通知系统架构评审报告
## 结论
这套方案能落地，但当前设计偏“链路堆叠”，不是“可稳定运营”的通知系统。
对 2 人团队、3 个月、1000 万/天峰值这个组合来说，最大风险不是单点吞吐，而是：重复投递、延迟堆积、外部依赖抖动、排障困难、缺少补偿机制。
如果不补齐幂等、告警、DLQ、状态机和可观测性，系统上线后大概率会变成“看起来都在跑，但没人敢说它可靠”。
---
## 一、核心问题
### 1. 链路过长，实时性被多次转发稀释
当前路径是：API Gateway → Lambda → DynamoDB → Streams → Lambda → SNS → SQS → Lambda → 外部服务。
这至少经过 6 次服务边界切换。
问题：
- 每一跳都增加延迟、重试和故障面。
- 任一环节异常都会放大排障复杂度。
- “实时通知”本应追求短链路，现在更像异步流水线。
建议：
- 先定义“必须多播”的场景，再决定是否保留 SNS。
- 如果只是按渠道分流，考虑直接路由到三个 SQS，少一层中转。

### 2. 没有看到明确的幂等设计
通知系统里，幂等不是优化项，是生存项。
问题：
- 移动端重试会造成重复创建。
- Streams、SQS、Lambda 都是至少一次语义。
- 下游服务临时错误重试后，重复发送风险很高。
建议：
- 入口请求必须带幂等键。
- 全链路携带 notificationId。
- 渠道投递前先查/写投递状态，避免重复发送。

### 3. DynamoDB 被当成“主存储 + 事件源 + 中转层”
问题：
- DynamoDB 适合高并发键值读写，不适合承担过多消息流转语义。
- 如果表设计不围绕查询模式，后面做历史查询、状态回查、按渠道筛选会很痛苦。
- 热点分区风险在通知类系统里很常见。
建议：
- 明确 DynamoDB 只做“通知主记录”和状态存储。
- 先定义查询模式，再定主键/排序键。
- 不要把它当消息总线。

### 4. Streams 触发器天然是至少一次，且可能批量重试
问题：
- 记录更新可能触发重复事件。
- 同一批次某条失败会拖累整批重试。
- 顺序性不稳定，容易出现状态错乱。
建议：
- 消费端按 notificationId 做幂等。
- 处理前先落投递任务状态。
- 对批失败、重复事件、乱序事件做专项测试。

### 5. SNS 放在中间层的收益不清晰
问题：
- 如果只有三个渠道队列，SNS 只是多一层转发。
- 多一层就多一组权限、多一组监控、多一组故障域。
- 出问题时，发布成功、订阅失败、队列积压会更难拆分。
建议：
- 若没有额外订阅者，考虑直接从分发 Lambda 写入三个 SQS。
- 若未来确实要广播给更多系统，再保留 SNS。

### 6. 每个渠道一个消费者没问题，但失败策略必须统一
问题：
- 三个 Lambda 的超时、批大小、并发、重试若不一致，行为会很难预测。
- 某个渠道慢会导致该队列积压，而其他渠道仍继续推进。
- 最终用户看到的是“部分到达、部分未到达”。
建议：
- 三个渠道统一错误分类、重试上限和退避策略。
- 每个队列都配 DLQ。
- 每个渠道单独记录投递结果。

### 7. 外部依赖是主要风险源
FCM、SES、短信网关都会限流、超时、返回临时错误。
问题：
- 外部抖动不是异常情况，而是常态。
- 如果没有退避和补偿，临时故障会变成积压风暴。
建议：
- 区分可恢复与不可恢复错误。
- 可恢复错误指数退避。
- 不可恢复错误直接进 DLQ 并标记原因。

## 二、容量与性能问题

### 1. 日均 1000 万不代表峰值安全

1000 万/天平均约 116 条/秒，但峰值可能远高于平均值。

问题：

- 峰值会同时打到 API Gateway、Lambda、DynamoDB、Streams、SNS、SQS 和下游接口。
- 只看日总量，会低估尖峰压力。

建议：

- 先明确峰值倍率。
- 对每一层做容量预算。

### 2. Lambda 冷启动和并发尖峰会放大延迟

问题：

- 链路里有多段 Lambda，任何一次冷启动都会传导到端到端时延。
- 高峰时如果并发没有规划，p95/p99 会明显恶化。

建议：

- 对关键消费者做压测。
- 明确超时、并发和批大小。

### 3. SQS 会掩盖问题，不会消灭问题

问题：

- 队列能缓冲瞬时峰值，但也会掩盖下游持续变慢的事实。
- 如果没人盯积压，通知延迟会在数小时后才被发现。

建议：

- 必须监控队列深度、最老消息年龄、消费者错误率。

## 三、数据一致性与业务语义问题

### 1. 没有定义通知状态机

至少应有：created、queued、delivering、delivered、failed、retrying。

问题：

- 没有状态机，就无法向业务解释当前通知在哪一步。
- 也无法做准确补偿和运营重放。

建议：

- 先定状态机，再定表结构和事件流。

### 2. 重复投递会直接伤害用户体验

问题：

- 同一用户收到两次推送、两封邮件、两条短信，在通知系统里属于事故，不是小瑕疵。

建议：

- 端到端幂等。
- 所有投递操作都写审计记录。

### 3. 顺序性需求没有被显式确认

问题：

- 某些业务要求同一对象或同一用户的通知有顺序。
- 该方案没有保证顺序的设计。

建议：

- 明确哪些场景要求顺序。
- 只对必要场景加约束，不要全局强求。

## 四、可观测性问题

### 1. 只用 CloudWatch 默认仪表盘不够

默认面板只能看到基础资源指标，看不到业务质量。

必须补的指标：

- 端到端投递成功率
- 分渠道成功率
- p95/p99 投递延迟
- DLQ 数量
- 队列最老消息年龄
- 外部服务错误率
- 重试次数和重复投递率

建议：

- 建业务指标面板，不要只看云资源面板。
- 所有关键指标设置告警阈值。

### 2. 缺少统一关联 ID 和链路追踪

问题：

- 多服务、多队列、多外部接口，没有统一 trace/correlation ID 会很难查。

建议：

- 每条通知生成统一 ID。
- 日志、指标、事件都透传这个 ID。

### 3. 没提告警策略和值班边界

问题：

- 没有告警，故障只能靠人发现。

建议：

- 至少要有 Lambda 错误率、DLQ、队列积压、外部服务错误和延迟告警。

---

## 五、成本与交付风险

### 1. 多层转发会抬高成本

问题：

- 每条通知经过多次服务调用和事件转发，成本会被层层叠加。

建议：

- 先算单条通知总成本。
- 能去掉的中间层尽量去掉。

### 2. 2 人团队不适合背太多运维复杂度

问题：

- 组件越多，测试、文档、监控、演练、排障工作越多。

建议：

- 优先缩短链路、减少故障域。
- 把可运维性当作上线门槛。

### 3. 没看到故障演练计划

建议上线前至少演练：

- 外部服务 5xx
- 队列堆积
- Lambda 超时
- 重复消息
- DLQ 回放

---

## 六、我会如何调整

### 推荐方向

1. 保留无服务器，但减少一层中间转发。
2. 把 DynamoDB 只定位为主记录和状态存储。
3. 为每个渠道建立独立 DLQ、告警和幂等控制。
4. 补齐业务级监控，而不是只看默认云监控。
5. 明确峰值假设后再做容量预算。

### 最低可上线标准

- 幂等键
- 状态机
- DLQ
- 告警
- 统一 correlation ID
- 失败重放方案

---

## 七、最终意见

这个架构的最大问题不是“不能用”，而是“上线后很难稳定地证明它一直在正确地用”。

如果你们坚持 3 个月内上线，我会优先做一个更短链路、更强幂等、更强观测的版本，而不是把所有托管组件都串起来。

否则最可能发生的故障不是完全宕机，而是：

- 重复通知
- 投递延迟越来越大
- 某个渠道长期失败却没人及时发现
- 出问题后无法快速补救

这类问题在通知系统里，比“单次发布失败”更危险。
