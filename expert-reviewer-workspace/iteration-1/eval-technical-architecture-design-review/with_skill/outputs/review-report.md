## 详细问题清单

### 问题 #1：跨库跨服务的"回滚"在分布式系统中不存在——分布式事务方案完全未指定

- **严重等级**：🔴 Critical
- **发现阶段**：Stage 3 ARGUMENTS
- **视角**：实现者 + 反对者
- **原文引用**：

  > "调用户服务扣减余额 → 调商品服务扣库存 → 更新订单状态为 SUCCESS，如果中间失败就回滚。"
  >
  > —— 用户原文（订单流程描述段）

- **方法论依据**：Toulmin 论证拆解 — Claim："如果中间失败就回滚"；Warrant（隐含）："跨服务跨库的操作可以原子回滚"。**该 Warrant 是错误的**：在没有 2PC / Saga / TCC / Outbox 的前提下，跨独立数据库不存在原子回滚原语。出处：`references/methodology-foundations.md` §3 Toulmin（Warrant 被默认的典型失败模式）。

- **🟥 红队视角**：

  假设我是双 11 当天的故障复盘工程师，这是必中招：

  - **场景 A**：用户服务扣款成功 → 网络抖动 → 商品服务调用超时 → 订单服务触发"回滚"——但用户服务这笔扣款在外部银行/支付通道已经走完，"回滚"怎么落实？钱已经从用户账户出去了。
  - **场景 B**：扣款成功 + 扣库存成功 → 订单库写 SUCCESS 时磁盘满 → 订单表是 PENDING 但余额已扣、库存已扣。用户看到 PENDING，钱没了，库存也没了。事后怎么发现？怎么恢复？
  - **场景 C**：商品服务返回成功但响应在网络中丢失 → 订单服务按"失败"触发"回滚"用户服务 → 实际上库存已扣且未被回滚 → 库存永远少一件。
  - 我会把这三条做成 PoC 写进漏洞报告，标题就叫《双 11 资损事件：分布式事务幻觉》。

- **🟩 建设性视角**：

  在订单流程那段后面，必须**强制**补一段"分布式事务策略"，从以下 4 种里选一种并说明理由：

  1. **TCC**：Try（冻结余额 + 预占库存）→ Confirm / Cancel。强一致，但要求所有服务改造。
  2. **Saga（编排式）**：每一步配一个补偿动作（退款 → 还库存 → 取消订单）。最终一致，业务侵入小。
  3. **本地消息表 + Kafka（Outbox）**：订单服务写订单 + 写本地消息表在一个本地事务里，异步发 Kafka 让用户/商品服务消费扣减。强最终一致，与现有 Kafka 选型契合。
  4. **事务消息（RocketMQ）**：与 #3 类似但用原生事务消息。

  每种方案都要列**正向 + 补偿**两个方向的接口定义和**超时阈值**。这是评审能放行的最低门槛。

### 问题 #2：没有任何幂等设计——重试/重放必然导致重复扣款与重复扣库存

- **严重等级**：🔴 Critical
- **发现阶段**：Stage 2 PERSPECTIVES（反对者视角，问题 1.5/1.6 输入验证、重放）
- **视角**：反对者/竞争对手（黑客视角）
- **原文引用**：

  > "订单服务创建订单（status=PENDING）→ 调用户服务扣减余额 → 调商品服务扣库存"
  >
  > —— 用户原文（订单流程描述段）。**未出现**任何幂等键、requestId、version、token 字眼。

- **方法论依据**：Perspective-Based Reading — 反对者视角预设问题 1.5/1.7（输入验证、关键数据可被切断）。出处：`assets/templates/perspective-questions.md` §视角 4。同时触发 `references/technical-architecture-review.md` §5 anti-pattern 嗅觉（"Sync chain"）。

- **🟥 红队视角**：

  我是攻击者，10 行 Python 就能搞垮系统：

  - **重放攻击**：抓一次合法下单请求，并发重放 1000 次。没有 requestId 去重 → 用户余额被扣 1000 次。双 11 当天 PR 灾难。
  - **网络重试副作用**：OpenFeign 默认有 retry；用户服务扣款成功但响应超时 → Feign 重试 → 扣第二次。这种事故在阿里/美团早期都发生过。
  - **客户端误点**：用户连点两次"提交订单"。前端没防抖 + 后端没幂等 → 两个订单。

  双 11 当天，上述任意一个都会直接上热搜。

- **🟩 建设性视角**：

  在订单流程图里，**第一步创建订单时就要生成全局唯一的 idempotency_key**（推荐 UUID v7 或雪花 ID），并在所有下游服务调用中携带。下游服务必须：

  - 用 `(user_id, idempotency_key)` 做唯一索引去重；
  - 重复请求返回第一次的结果，不重复执行；
  - 数据库层用乐观锁（version 字段或 `UPDATE balance SET balance = balance - ? WHERE id = ? AND balance >= ?`）。

  同时所有 OpenFeign 调用必须配置**只对非幂等接口禁用自动重试**，或显式改写为幂等接口后才允许重试。

### 问题 #3：库存必然超卖——单行 UPDATE 在双 11 并发下撑不住，且未提任何热点防护

- **严重等级**：🔴 Critical
- **发现阶段**：Stage 4 DECISIONS（Pre-mortem #1）
- **视角**：反对者 + 决策者
- **原文引用**：

  > "调商品服务扣库存"
  >
  > —— 用户原文。**未出现** Redis 预扣、库存分桶、热点 key、version、select for update 等任何字眼。

- **方法论依据**：Pre-mortem（Klein 2007）——假设方案已失败，最可能原因之一是"库存超卖大爆发"。出处：`references/bias-checks.md` §1。同时触发 `references/technical-architecture-review.md` §1 ATAM——双 11 是显式质量属性场景但未构造具体场景（如"单品 10 万件秒杀，1 秒内 100 万次扣减请求"）。

- **🟥 红队视角**：

  我是黄牛，双 11 0:00:00 我用 5000 个并发请求打茅台库存：

  - **MySQL 单行 UPDATE 是行锁串行**：同一行 `UPDATE stock SET qty = qty - 1 WHERE sku = ? AND qty > 0` 在 InnoDB 下走行锁，5000 QPS 即开始出现锁等待超时。
  - **跨服务调用增加了窗口**：你的扣库存是在"商品服务"里，前面还要先经过 Gateway → 订单服务 → 用户服务 → 商品服务 4 跳 RTT，锁持有时间被拉长 10 倍。
  - **Redis 与 MySQL 的一致性窗口**：如果用 Redis 缓存库存（你没说），多线程下扣减 Redis 与回写 MySQL 之间存在窗口 → 经典超卖。

  双 11 当晚会有 1000 个用户付款成功但订单显示"库存不足"，资损 + 投诉 + 监管。

- **🟩 建设性视角**：

  必须在文档里**显式说明库存扣减的层次**：

  1. **Redis 预扣库存**（原子 `DECR` / Lua 脚本）作为第一道闸门；不足直接拒绝。
  2. **数据库层**用 `UPDATE ... WHERE qty >= ?` 做兜底，**禁止** `SELECT ... FOR UPDATE` 长事务。
  3. **热点商品库存分桶**（如茅台库存 1000 件拆成 100 个桶，每桶 10 件，路由到不同行）。
  4. **异步落库**：Redis 扣减后写 Kafka，商品服务消费异步更新 MySQL。
  5. **对账脚本**：每分钟跑一次 Redis vs MySQL 差异告警。

  不补这一段，方案不能放行。

### 问题 #4："支撑双 11 大促"——零量化指标、零容量规划、零场景

- **严重等级**：🔴 Critical
- **发现阶段**：Stage 2 PERSPECTIVES（决策者视角）+ Stage 4（Outside-In / Reference Class）
- **视角**：决策者/投资人
- **原文引用**：

  > "设计目标是支撑双 11 大促。"
  >
  > —— 用户原文最后一句。**全文未出现**任何 QPS、RT、SLA、容量、并发数等数字。

- **方法论依据**：ATAM 场景驱动评审 — "每个 quality attribute 是否有可量化的目标（如 P99 < 100ms）"。出处：`references/technical-architecture-review.md` §1（典型问题清单）。同时触发 Outside-In Thinking（`references/bias-checks.md` §8）——"双 11 大促"的同类系统（淘宝、京东、拼多多）真实 QPS 是公开数据，本方案与 reference class 完全没有对照。

- **🟥 红队视角**：

  我是评审委员会，"支撑双 11 大促"这句话不构成可验收目标。

  - 淘宝双 11 峰值订单创建速率公开数据约 **58.3 万笔/秒**（2020）。
  - 一个普通中型电商双 11 峰值至少 **1-5 万 QPS**。
  - 你的方案在哪种量级下成立？1 千 QPS？1 万？10 万？—— 文档没说。
  - 没有数字就无法做容量规划：MySQL 用 8C16G 还是 64C256G？Redis 单点还是集群？Kafka 几个分区？OpenFeign 连接池多大？全部是空中楼阁。

  上线前一周才会发现容量假设错了——这是双 11 故事的经典开篇。

- **🟩 建设性视角**：

  把"双 11 大促"换成具体场景表（ATAM 风格）：

  | 场景 | 目标 | 验收方法 |
  |---|---|---|
  | 0:00 峰值下单 | 峰值 X 万 QPS，P99 < 500ms，错误率 < 0.01% | 全链路压测 |
  | 单 AZ 宕机 | 5 分钟内切换，RTO < 5min，RPO < 1min | Chaos 工程 |
  | 单服务降级 | 库存服务挂，订单仍可创建（待处理） | 故障注入 |
  | 数据一致性 | 任意时刻三库差异 < 10 笔/分钟 | 对账脚本 |

  X 由业务方提供，但**必须在文档里出现**。

### 问题 #5：同步调用链无熔断/限流/超时配置——级联雪崩不可避免

- **严重等级**：🔴 Critical
- **发现阶段**：Stage 4 DECISIONS（Pre-mortem #3）
- **视角**：维护者 + 反对者
- **原文引用**：

  > "调用户服务扣减余额 → 调商品服务扣库存"
  >
  > —— 用户原文。使用 OpenFeign 同步调用，**全文未出现** Sentinel / Hystrix / Resilience4j / 限流 / 熔断 / timeout。

- **方法论依据**：Well-Architected Framework Reliability 支柱（"故障恢复时间目标？降级路径？"）。出处：`references/technical-architecture-review.md` §4。同时触发 anti-pattern 嗅觉"Sync chain"（同上 §5）。

- **🟥 红队视角**：

  双 11 0:01，商品服务的 MySQL 突然变慢（磁盘 IO 飙高），从 5ms 涨到 500ms：

  - 订单服务 OpenFeign 调用商品服务的请求堆积（默认无超时或 60s 超时）；
  - 订单服务 Tomcat 线程池（默认 200）30 秒内耗尽；
  - 订单服务无法接收新下单请求；
  - Gateway 也开始堆积；
  - React 前端超时 → 用户疯狂重试 → 流量翻倍；
  - 用户服务的连接池也开始堆积；
  - **5 分钟内全栈瘫痪**。

  这是分布式系统教科书级的级联雪崩。阿里 2008 年淘宝早期就吃过这个亏。

- **🟩 建设性视角**：

  每一次跨服务调用都必须显式配置：

  1. **超时**：connectTimeout=200ms，readTimeout=500ms（双 11 严苛场景）。
  2. **熔断**：Sentinel 或 Resilience4j，错误率 > 1% 触发熔断 10s。
  3. **限流**：订单服务入口 QPS 限流（令牌桶），用户服务/商品服务均自限流。
  4. **舱壁隔离**：不同下游用独立线程池，避免相互拖垮。
  5. **降级策略**：商品服务熔断时返回"系统繁忙"，订单进入"待处理"队列而非直接失败。

  这些都是 Spring Cloud Alibaba Sentinel 的标准用法，必须写进文档。

### 问题 #6：Kafka 的角色未定义——同步流程中找不到它的位置

- **严重等级**：🟠 High
- **发现阶段**：Stage 1 PARSE（一致性检查）+ Stage 3 ARGUMENTS
- **视角**：实现者 + 维护者
- **原文引用**：

  > "消息队列 Kafka" 在架构清单里出现；
  >
  > 但 "订单流程是：用户下单 → 订单服务创建订单 → 调用户服务扣减余额 → 调商品服务扣库存 → 更新订单状态为 SUCCESS，如果中间失败就回滚"——**整个流程是同步的，Kafka 无处出现**。
  >
  > —— 用户原文（架构段 vs 流程段对比）

- **方法论依据**：Cognitive Dimensions — Consistency（"同一概念在不同地方用法一致吗？"）。出处：`references/technical-architecture-review.md` §3。同时触发 ADR 反向审计——"我们用 Kafka"无 alternatives（为什么不是 RocketMQ / Pulsar / RabbitMQ？）。

- **🟥 红队视角**：

  我是新入职的工程师，看完文档问："Kafka 用来干嘛？"——没人答得上来。两种可能性都糟糕：

  - **如果 Kafka 真的没用**：架构图是装饰，团队需要维护一个完全没必要的中间件，浪费资源 + 引入额外故障点。
  - **如果 Kafka 有用但没写**：那它最该出现在分布式事务里（Outbox 模式）或异步解耦（下单后发短信/积分/物流），而这两个用法都**与同步流程矛盾**——文档同时描述了"同步调用 + 回滚"和"有 Kafka"，二者必有一处是错的。

- **🟩 建设性视角**：

  二选一并明确：

  - **方案 A（推荐）**：把 Kafka 用作 **Outbox 模式的消息总线**，订单本地事务 + 本地消息表 → Kafka → 下游消费。这与"双 11 大促"的目标契合，可以削峰。
  - **方案 B**：保留同步流程，**删除 Kafka**，避免引入未使用的复杂度。

  不允许"既有 Kafka 又不用"的状态。

### 问题 #7：所有技术选型无 ADR——为什么 Spring Cloud 不是 Dubbo？为什么 Kafka 不是 RocketMQ？

- **严重等级**：🟠 High
- **发现阶段**：Stage 3 ARGUMENTS（Toulmin Backing 缺失）
- **视角**：实现者 + 决策者
- **原文引用**：

  > "前端 React + 后端 Spring Cloud（Gateway + Nacos + OpenFeign）+ 数据库 MySQL（订单服务一个库、用户服务一个库、商品服务一个库）+ 消息队列 Kafka + 缓存 Redis。"
  >
  > —— 用户原文。**所有选型都是断言，无 alternatives，无 context，无 consequences**。

- **方法论依据**：ADR 反向审计——"任何决策缺 alternatives → High 严重等级"。出处：`references/technical-architecture-review.md` §2。

- **🟥 红队视角**：

  我是接手维护这份系统的工程师，3 年后这些选型的语境可能完全变了：

  - 为什么 Kafka 不是 RocketMQ？阿里双 11 实战都靠 RocketMQ 的事务消息，Kafka 在严格事务语义上不如 RocketMQ。
  - 为什么 Spring Cloud 不是 Dubbo？Dubbo 在国内高性能 RPC 场景下吞吐更高。
  - 为什么三个独立库？订单/用户/商品的业务耦合度其实很高（下单同时涉及三者），独立库引入分布式事务难题，是否值得？
  - 为什么 MySQL 不是 TiDB / PolarDB？后者原生分布式，可避免本方案最致命的分布式事务问题。

  这些问题没答案，意味着选型是"凭感觉"而非"基于约束"。

- **🟩 建设性视角**：

  为每一个关键选型补一份 ADR（哪怕 200 字一段）：

  ```
  ADR-001: 选择 Spring Cloud 而非 Dubbo
  Context: 团队 Java 背景，需要 Gateway + 服务发现 + 配置中心一体化
  Alternatives: Dubbo + Nacos（更高性能但生态窄）、gRPC + 自建网关
  Consequences: 性能略低，但开发效率高，团队学习成本低
  Compliance: 双 11 压测验证 RT < 500ms
  ```

### 核心论证的 Toulmin 拆解

#### 论证 #1：用同步调用 + "回滚" 能支撑双 11 大促

| 要素 | 内容 | 评审 |
|---|---|---|
| Claim 结论 | "支撑双 11 大促" | ❌ 模糊到无法证伪（无 QPS 数字） |
| Data 数据 | 仅"前端 React + 后端 Spring Cloud + MySQL×3 + Kafka + Redis" | ❌ 不构成支撑 Claim 的数据 |
| Warrant 推理依据 | （隐含）"主流技术栈 + 同步调用 + 回滚"足以支撑双 11 | ❌ Warrant 错误——见论证 #2 |
| Backing 支撑 | 缺失——没有"同类系统用这套方案支撑了双 11"的 reference class | ❌ 完全缺失 |
| Qualifier 限定词 | 无（"能支撑"是绝对断言） | ❌ 过度绝对 |
| Rebuttal 反例 | 缺失——未承认任何反例（如同步链雪崩、超卖、对账难题） | ❌ 完全缺失 |

**Walton critical questions**（实践型论证）：

- "同步调用 + 回滚" 真的能达成"支撑双 11"？→ 不能（见问题 #1 #3 #5）
- 副作用是否抵消价值？→ 是（资损、超卖、雪崩）
- 是否存在更好的行动（Saga / TCC / Outbox）？→ 是，且文档未考虑

#### 论证 #2：如果中间失败就回滚

| 要素 | 内容 | 评审 |
|---|---|---|
| Claim 结论 | 失败后系统能回到一致状态 | ❌ Claim 错误 |
| Data 数据 | 无 | ❌ |
| Warrant 推理依据 | （隐含）"回滚"是一个原子操作 | ❌ **Warrant 错误**——跨独立数据库无原子回滚原语 |
| Backing 支撑 | 缺失 | ❌ |
| Qualifier 限定词 | 无（"就回滚"是绝对断言） | ❌ |
| Rebuttal 反例 | 缺失 | ❌ |

**Walton critical questions**（因果型）：

- "回滚" 与 "一致" 真的有因果关系？→ 没有，前提条件（原子性）不成立
- 是否有第三方变量？→ 是：网络分区、超时、磁盘满、进程崩溃，任何一个都会让"回滚"本身失败
- 因果方向是否反了？→ 是：不是"失败→回滚→一致"，而是"失败→可能回滚→仍可能不一致"

#### 论证 #3：三个服务各一个 MySQL 库（database-per-service）

| 要素 | 内容 | 评审 |
|---|---|---|
| Claim 结论 | 这是合理的微服务设计 | ⚠️ 结论方向可能对，但缺 justification |
| Data 数据 | "订单服务一个库、用户服务一个库、商品服务一个库" | ⚠️ 仅描述，未给理由 |
| Warrant 推理依据 | （隐含）database-per-service 是微服务最佳实践 | ⚠️ 是常见实践，但**非唯一**，也非无成本 |
| Backing 支撑 | 缺失——未引用 Fowler / Newman 等微服务文献 | ❌ |
| Qualifier 限定词 | 无 | ❌ |
| Rebuttal 反例 | 缺失——未承认代价（分布式事务难题、对账复杂度、调试难度） | ❌ |

**Walton critical questions**（专家意见型）：

- "database-per-service" 真的是该场景最佳？→ 不一定，业务耦合度高的下单场景反而增加复杂度
- 其他专家是否同意？→ 业界对"高耦合业务是否要分库"有争议，文档未承认
- 文档引用是否被准确转述？→ 无引用

### ACH（竞争假设分析）

文档主张："**应该用同步调用 + 本地回滚** 实现订单流程"。

竞争假设：

- **假设 A（文档）**：同步调用 + 本地"回滚"
- **假设 B**：TCC（Try-Confirm-Cancel）
- **假设 C**：Saga（编排式 + 补偿事务）
- **假设 D**：本地消息表 + Kafka Outbox + 最终一致

| 证据 \ 假设 | A（文档） | B（TCC） | C（Saga） | D（Outbox） |
|---|---|---|---|---|
| 文档已有 Kafka | 一致（未用） | N/A | 一致 | ✅ **强一致** |
| 文档已有三独立库 | 一致 | 一致 | 一致 | 一致 |
| 双 11 峰值削峰需求 | ❌ 不一致 | ❌ 不一致 | ⚠️ 部分 | ✅ **强一致** |
| 业务方接受最终一致 | 未知 | N/A | 一致 | 一致 |
| 开发成本 | 最低 | 最高 | 中 | 中 |
| 一致性强度 | ❌ **最弱（实际无保证）** | ✅ 最强 | 中 | 中 |
| **诊断性** | A 的"低成本"在"双 11 + 一致性"前提下不成立 | — | — | D 与 A 在"已有 Kafka"和"双 11 削峰"上诊断性最强 |

**结论**：文档完全没有比较 A/B/C/D，且 A 在双 11 场景下属于"低成本但不成立"的选择。**最低 High，建议 Critical**（与问题 #1 一并算 Critical）。

## 评审者自评

### 本份评审**没有覆盖到**的方面

- **代码层面**：文档是口述，无代码可审。代码层面的 SQL 注入、序列化漏洞、依赖 CVE 等未覆盖。
- **运维细节**：监控大盘的具体指标、SLO/SLI 体系、On-call 流程未深入。
- **业务侧**：退款流程、对账频率、风控规则、客服侧工具未展开。
- **数据治理**：用户数据 GDPR/个保法合规、数据生命周期管理未深入。
- **成本估算**：双 11 弹性扩容的成本模型、长期 TCO 未计算。
- **法务/合规扩展视角**：未深入展开（只在问题 #10 触及）。

### 本份评审**存在偏倚**的可能来源

- **严苛偏倚**：用户明确说"越严越好"，评审者可能倾向于过度挑刺。但本评审的 Critical 都基于双 11 同类系统的真实历史故障（淘宝、京东早期公开案例），偏倚可接受。
- **技术栈偏好**：评审者倾向"Outbox + 最终一致"，可能低估 TCC/2PC 在特定场景下的价值。已在 ACH 中尝试平衡。
- **规模假设**：假设用户的"双 11"指中型电商级别（1-5 万 QPS）。如果是 100 QPS 级别的小电商，部分 Critical 可降为 High。**这一点请用户校准**。
- **文档形态偏倚**：评审针对的是 130 字口述，而非完整设计文档。完整设计文档可能已隐式处理了部分问题，评审无法看到。

### 如果只能改一件事

> **把"如果中间失败就回滚"这句话换成具体的分布式事务方案**（推荐本地消息表 + Kafka Outbox）。这一改会强制回答：扣款成功后订单写库失败，怎么退钱？——这个问题答清楚了，方案就活了；答不清楚，双 11 必然翻车。
